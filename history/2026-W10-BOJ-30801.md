# 2026-W10-BOJ-30801

## 문제
- 출처: BOJ
- 이름: 가희와 클럽 오디션 2
- 링크/번호: 30801

## 요약
- 목표: 노트 문자열과 입력 문자열을 방향으로 해석한 뒤, 모든 입력을 처리했을 때 최종 판정이 성공인지 판단한다.
- 까다로웠던 점: 문제 설명이 길어서 처음에는 "변환된 두 배열이 완전히 같으면 된다"로 과도하게 단순화하기 쉬웠다.

## 입출력 정리
- 입력: 첫 줄에 노트 문자열, 둘째 줄에 실제 입력 문자열이 주어진다.
- 출력: 최종 판정이 성공이면 `Yes`, 실패면 `No`.
- 제약: 노트 레벨은 `6` 이상 `11` 이하, 입력 길이는 `20000` 이하.

## 논의 기록
- 처음 관찰한 내용: 노트 토큰 파싱과 `!` 역방향 처리가 중요해 보였다.
- 사용자 주도 방향: 먼저 문제 해설을 정리하고, 그 내용을 `plan.md`에 남긴 뒤 구현을 맞춰 가는 방식으로 진행했다.
- 버린 아이디어:
  - 변환된 노트 방향 배열과 입력 방향 배열이 길이와 내용까지 완전히 같으면 성공이라고 해석했다.
  - `matched == lv` 상태에서 추가 입력이 들어왔을 때, 그 입력으로 다시 첫 노트와 비교해도 된다고 잘못 처리했다.
- 최종 선택 방향과 이유:
  - 상태 시뮬레이션으로 정리했다.
  - 상태는 `앞에서부터 몇 개를 연속으로 맞춘 상태인가` 하나만 유지하면 충분했다.
  - 문제 본문에 적힌 전이 규칙을 그대로 구현해야 반례를 놓치지 않는다.

## 구현 메모
- 파싱 결정:
  - 노트는 `W`, `A`, `S`, `D`, `LU`, `LD`, `RU`, `RD` 토큰으로 끊고, `!`가 붙으면 반대 방향으로 바꿨다.
  - 입력은 각 문자를 방향 상수로 변환했다.
- 핵심 상태 / 자료구조:
  - `note []int`
  - `keys []int`
  - `matched int`
- 중요한 엣지 케이스:
  - 앞에 불필요한 입력이 있어도 마지막에 정확히 맞춰 끝나면 성공할 수 있다.
  - 이미 전부 맞춘 상태에서 다음 입력이 들어오면, 그 입력은 리셋만 일으키고 같은 입력으로 다시 비교하면 안 된다.

## 검증
- 확인한 예제:
  - 예제 1 `Yes`
  - 예제 2 `No`
  - 예제 3 `No`
  - 예제 4 `No`
- 추가로 확인한 케이스:
  - `DDDDDD / ADDDDDD` -> `Yes`
  - `D / DD` -> `No`
- 최종 결과:
  - 잘못된 단순 비교 해석을 버리고 상태 시뮬레이션으로 수정한 뒤 테스트를 통과했다.

## 회고
- 다음에도 재사용할 점:
  - 상태 전이 규칙이 문제 본문에 명시되어 있으면, 지나치게 단순화하기 전에 반례를 먼저 점검한다.
  - 오답이 나오면 "내 해석이 문제 규칙을 삭제한 건 아닌지"를 먼저 본다.
- 개선할 점:
  - 이번에는 중간에 `plan.md` 해석을 잘못 수정했다. 다음에는 반례를 찾기 전까지 핵심 판정 규칙을 너무 빨리 단순화하지 않는다.
